# 学園自治シミュレーター 初期仕様メモ

## 1. 目的

仮想の学園運営を題材に、生徒・教師・保護者など複数ステークホルダーの声を集め、AIで論点化し、施策を決め、その結果をターン制で可視化するWebアプリケーションを作る。

主目的は、単純な人気取りではなく、民主的な合意形成と公共的な帰結のズレを体験できること。

## 2. MVPの舞台

- 舞台: 仮想学園
- プレイヤー: 生徒会
- 1ターン: 1学期
- 初期テーマ: 固定しない。初期指標と仮想ペルソナの声からAIが課題候補を抽出する
- データ: 実在個人ではなく、仮想指標データと仮想ペルソナを使う
- 初期固定データ: JSONで管理する

## 3. 1ターンの流れ

1. 学期開始時に、初期指標または前学期結果から現在の学園状態を作る
2. 学園状態には乱数シードに基づく揺らぎを入れる
3. 生徒・教師・保護者などの仮想ペルソナが、現在の状態を見て不満・要望・懸念を発言する
4. AIが発言を分類、要約し、課題候補を抽出する
5. 生徒会がAIとのチャットを通じて課題候補を深掘りし、今学期に取り上げる課題を選ぶ
6. AIとユーザーがチャットで改善案を複数検討する
7. ユーザーが施策、キャッシュ使用額、必要に応じた予算充当案を決める
8. AIが施策結果のシミュレーションを行い、指標変更案、理由、副作用、ランダム要素を出す
9. アプリ側がAI出力を検証し、見えるスコアと見えないスコアに反映する
10. 指標変化、属性別影響、副作用、次学期への論点を可視化する
11. 次ターンへ進む

初回ターンでは課題を固定表示しない。まず学園の初期状態を生成し、その状態に対するペルソナ発話から課題候補を作る。

## 4. 固定する中核要素

以下はプロダクトの核として固定する。

- ターン制
- 学園全体指標
- 属性別指標
- 初期指標のランダム揺らぎ
- 仮想ペルソナからの多様な声
- AIによる課題候補の抽出
- AIによる論点整理と改善案検討
- 施策の短期効果と長期効果
- 施策の副作用
- ランダムイベント
- 結果の見える化

## 5. 指標案

### 学園全体指標

- 予算
- キャッシュ
- 運営費
- 生徒全体支持率
- 教師支持率
- 保護者支持率
- 学校評判
- 成績
- 幸福度
- 運営安定度
- 校則納得度
- 不公平感
- 参加率
- 教師負担

### 見えるスコアと見えないスコア

ゲーム性を高めるため、すべての状態を常時ダッシュボードに出さない。

#### 見えるスコア

生徒会が通常の運営で把握できる情報。

- 予算
- キャッシュ
- アンケート結果から見える支持率
- 生徒から見える評判
- 成績
- 参加率
- 校則納得度
- 集まった意見の傾向
- 選択施策の予測効果

#### 見えないスコア

通常ダッシュボードでは直接表示しない内部評価。

- 信頼の蓄積
- 分断リスク
- 制度疲労
- 少数派の潜在的不満
- 教師側の運用限界
- 長期公共性
- 施策と説明の一貫性

見えないスコアは、施策結果、発話傾向、ガードレール判定、短期成果と長期成果のズレから内部的に変動する。通常のターン中は数値を出さず、年度末レポートやイベントとして間接的に見せる。

### 属性別指標

- 自由重視層
- 受験重視層
- 部活重視層
- 文化活動重視層
- 校則重視層
- 経済的に厳しい層
- 無関心層
- 改革派
- 慎重派

各属性は人数比、関心テーマ、感受性、発言頻度、投票行動、施策への反応係数を持つ。

### 初期指標のランダム性

初回開始時に、固定JSONの基準値から一定範囲で指標を揺らす。

例:

```json
{
  "metricId": "studentSupport",
  "base": 55,
  "variance": 8,
  "min": 0,
  "max": 100
}
```

乱数はシード管理する。同じ保存データを読み込めば同じ状態を再現できるようにする。

## 6. AIの役割

AIは意思決定の代行者ではなく、熟議支援者として扱う。

### AIが行うこと

- 仮想ペルソナに基づく意見生成
- 意見の分類
- 要約
- 課題候補の抽出
- 論点抽出
- 対立軸の整理
- 少数意見の検出
- 改善案の生成
- 施策の影響仮説の作成
- 結果レポートと次ターン論点の作成

### AI中心の前提

本アプリはAIなしで成立するゲームではない。AIが仮想データを生成し、声を再現し、課題を抽出し、ユーザーとの対話を通じて施策を深掘りすることが中核体験である。

`local-mock` は、AIなしで遊ぶための主要モードではなく、開発中の固定サンプル、テスト、AI障害時のfallbackとして扱う。

### AI Provider方針

公開Webでは利用者自身のAPIキー入力を採用し、運営側のAPIキーは持たない。APIキーは保存ファイルに含めず、ブラウザセッション内だけで扱う。

ProviderはOpenAI固定にしない。初期対応は以下とする。

- `openai_responses`: OpenAI Responses API
- `openai_compatible_chat`: OpenAI互換Chat Completions API
- `codex_app_server`: ローカル実行専用のCodex App Server接続
- `sample`: 開発用固定サンプルfallback

`codex_app_server` はローカル専用で、`codex app-server --listen ws://127.0.0.1:45123` を別プロセスで起動し、ブラウザからWebSocketで接続する。公開Web上では利用前提にしない。

### AIが行わないこと

- 最終施策を勝手に決める
- 正解を断定する
- 少数派の不利益を消す
- 多数決だけで合意形成を扱う

### 課題抽出のAI処理

AIは、学園状態とペルソナ発話をもとに、今学期に扱うべき課題候補を複数出す。

課題候補には以下を含める。

- 課題名
- 背景
- 関連する指標
- 強く反応している属性
- 少数派の懸念
- 放置した場合のリスク
- 生徒会で扱いやすい度合い
- 必要予算の大まかなレンジ

生徒会プレイヤーは、AIが抽出した課題候補から1つ以上を選ぶ。AIは候補を出すだけで、採択はユーザーが行う。

### 課題選択チャット

課題候補が出た時点からチャットを開始する。

ユーザーは、AIに以下を質問しながら今学期に扱う課題を絞り込む。

- どの課題が今学期に扱うべきか
- 放置した場合の影響が大きい課題はどれか
- 生徒会で実行可能な範囲はどこまでか
- 少数派に強く影響している課題はどれか
- 予算とキャッシュの範囲で現実的な課題はどれか
- 複数課題をまとめて扱える切り口はあるか

AIは推奨を出してよいが、最終的にどの課題を扱うかはユーザーが決める。

### 施策検討チャット

施策案は単発生成で終わらせない。

選択した課題に対してAIが分析レポートを出し、ユーザーはチャットで以下を深掘りする。

- なぜその課題が重要なのか
- どの属性にどのような影響が出るのか
- 少数派の懸念は何か
- 予算内で実行できる案は何か
- 人気は出るが長期的に危険な案は何か
- ガードレール上の懸念は何か
- 代替案や折衷案は何か

チャットを通じて施策ドラフトを更新し、最終的に1つの施策として確定する。

チャット文脈には、現在の学園状態、見えるスコア、生徒発話、課題候補、ガードレール方針、予算制約、キャッシュ制約を含める。見えない内部スコアの具体値は通常は渡さず、必要に応じて「潜在的なリスク傾向」として扱う。

### 施策結果シミュレーション

施策実行後の結果シミュレーションにもAIを使う。

AIは以下を出力する。

- 施策が各属性に与えた影響
- 見えるスコアの変更案
- 見えないスコアの変更案
- 予測と実績の差
- 発生したランダムイベント
- 短期的な成果
- 長期的な副作用
- 次学期に残る論点

アプリ側はAIの出力をそのまま無制限に反映しない。JSON Schemaで検証し、指標ごとの増減幅、予算、キャッシュ、ガードレールに反する変更がないかを確認してから反映する。

シミュレーション粒度は初期MVPでは粗く始め、実際の結果を見ながら調整する。

初期粒度:

- 施策単位
- 属性グループ単位
- 見えるスコア単位
- 見えないスコア単位
- ランダムイベント単位

将来的な粒度:

- ペルソナ単位
- 複数ターンにまたがる遅延効果
- 予算流用による副作用
- 教師・保護者など外部ステークホルダーへの影響

## 7. AI接続方針

AI接続はプロバイダー抽象化する。

### 公開Web利用

- OpenAI、Anthropic、Google Gemini、OpenAI互換APIなど、複数Providerの外部APIキーを利用できる設計にする
- APIキー運用は `bring-your-own-key` とする
- 運営側はAPIキーを提供しない
- ユーザーが画面でAPIキーを入力し、ブラウザのセッション内だけで保持する
- サーバー側にユーザーAPIキーを永続保存しない
- 公開環境では `codex-cli` モードを無効化する

公開サービスとして不特定多数に触らせる場合、運営者APIキーを使う `server-key` 方式は採用しない。初期MVPでは、ユーザー自身のAPIキー入力を前提にする。

### Provider方針

AI Providerは固定しない。

初期実装では以下から始める。

- `sample`: 開発用固定サンプル
- `openai_responses`: OpenAI Responses API
- `openai_compatible_chat`: OpenAI互換のChat Completions API

`openai_compatible_chat` はBase URLを差し替えられるようにし、OpenRouter、Groq、Togetherなどの互換APIを試しやすくする。

Anthropic、Google Geminiなど、専用のAPI形式が必要なProviderは、同じProviderインターフェースに追加する。

### ローカル利用

以下のモードを切り替え可能にする。

- `openai_responses`: OpenAI Responses APIを使う
- `openai_compatible_chat`: OpenAI互換Chat Completions APIを使う
- `anthropic`: Anthropic APIキーを使う
- `gemini`: Google Gemini APIキーを使う
- `local-mock`: 開発中の固定サンプル、テスト、AI障害時fallbackとして固定レスポンスを使う
- `codex-cli`: ローカル開発時のみ、サーバー側から `codex exec` を子プロセスとして呼び出す
- `cli-assist`: CodexやClaude Codeなど、ローカル開発時の支援AIで生成したJSONを読み込む

注意: CodexやClaude Codeの認証トークンをWebアプリが直接読み取ったり、ブラウザへ渡したりする設計にはしない。`codex-cli` はローカルのNext.jsサーバー上でのみ有効にし、公開環境ではビルド時または実行時設定で必ず無効化する。

### Codex CLI連携の考え方

この環境では `codex exec` による非対話実行が利用できるため、ローカル開発では以下の流れにできる。

1. Web画面から「意見生成」「論点整理」「施策案生成」などのAI処理を要求する
2. Next.js Route Handlerがローカル環境かつ `AI_PROVIDER=codex-cli` であることを確認する
3. サーバー側で `codex exec --output-schema <schema> <prompt>` を実行する
4. Codex CLIの最終出力をJSONとして検証する
5. Webアプリへ構造化データとして返す

この方式なら、WebアプリはCodexの認証トークンを扱わず、既にログイン済みのCodex CLIに処理を委譲できる。ただし、CLI起動は通常のAPI呼び出しより遅く、同時実行や公開運用には向かない。そのため本番はAPIキー方式、ローカル検証は `codex-cli` 方式と分ける。

`codex-cli` は正式対応候補に含める。ただし、初期MVPの必須条件にはしない。まずAI Provider設定と、開発用固定サンプルの入出力検証を安定させ、CLI実行の入出力・タイムアウト・エラー処理が整理できた段階で正式対応に昇格する。

### AI出力の検証

AI出力は必ずJSON SchemaまたはZodで検証する。

- 意見生成: `Opinion[]`
- 論点整理: `DeliberationSummary`
- 施策案生成: `PolicyOption[]`
- 結果分析: `TurnResultReport`

検証に失敗した場合は、再生成、エラー表示、または開発用固定サンプルへのfallbackを行う。

## 8. 保存方針

公開Webでは、基本的にアクセス時は新規開始とする。サーバー側にユーザー別の永続データを保存しない。

### ブラウザ内保存

- 現在のシミュレーション状態はブラウザ内に保存する
- 候補は `IndexedDB`
- 小さな設定値のみ `localStorage` を使う
- ブラウザキャッシュ削除や別端末アクセスでは復元されない前提にする

### メモリー設計

本アプリのメモリーは、AIの会話履歴だけではなく、シミュレーション状態の時系列ログとして扱う。

保存するメモリーは以下の4層に分ける。

- `currentState`: 現在の学期状態。ダッシュボード表示、次のAI入力、施策実行判定に使う
- `timeline`: 学期ごとのスナップショット。指標、予算、キャッシュ、属性別状態の推移を保持する
- `eventLog`: 施策実行、AIシミュレーション結果、ランダムイベント、ガードレール判定などの出来事を保持する
- `memorySummary`: 長期履歴をAIに渡しやすい短い要約。全履歴を毎回プロンプトに入れないために使う

#### `currentState`

現在ターンで画面とルールエンジンが直接参照する状態。

- 現在学年・年度
- 現在学期
- 乱数シード
- 見える指標
- 見えない内部スコア
- 予算
- キャッシュ
- 属性別意識状態
- 今学期の課題候補
- 選択中の課題
- 作成中の施策ドラフト

#### `timeline`

推移グラフ、年度末レポート、AIの長期文脈に使う履歴。

1学期ごとに以下を保存する。

- 学期ID
- 開始時スナップショット
- 終了時スナップショット
- 見える指標の増減
- 見えない内部スコアの増減
- 予算・キャッシュの増減
- 属性別意識の増減
- 主要な課題
- 採用施策
- AIが説明した因果仮説

#### `eventLog`

後から検証可能にするための追記型ログ。

- 生徒ペルソナ発話生成
- 課題候補抽出
- 課題選択チャット
- 分析レポート生成
- 施策ドラフト更新
- ガードレール判定
- 施策採用
- 施策結果シミュレーション
- ランダムイベント
- 年度末レポート生成

イベントには、`id`、`turnId`、`type`、`createdAt`、`summary`、`payload` を持たせる。AIの長文全文を常に保持すると保存ファイルが肥大化するため、MVPでは本文と要約を分け、必要に応じて本文を圧縮または削除できる設計にする。

#### `groupMemory`

各属性の意識遷移を保存する。

- 属性ID
- 現在の支持度
- 現在の不満度
- 信頼度
- 発言活性
- 主要関心テーマ
- 前学期からの変化
- その変化を引き起こした施策・イベント
- 次学期に表面化しそうな懸念

属性別意識は、単なる数値だけではなく「なぜ変化したか」の短い説明を持たせる。これにより、次ターンのAI発話が過去施策の影響を踏まえられる。

#### `memorySummary`

AIに渡すための圧縮メモリー。

全ターンの詳細ログを毎回AIに渡すとプロンプトが肥大化するため、以下の要約を保持する。

- これまでの運営方針の傾向
- 成功した施策
- 副作用が残った施策
- 属性別に蓄積している不満
- 信頼を得た理由
- 信頼を失った理由
- 次ターンでAIが考慮すべき制約

`memorySummary` は各ターン終了時と年度末にAIまたはルールエンジンで更新する。ただし、内部スコアの具体値をプレイヤーへ常時表示しない方針は維持する。

### ブラウザ保存方式

MVPでは `IndexedDB` を主保存先にする。

- `localStorage`: 最後に開いた保存スロットID、UI設定、Provider種別などの軽量設定
- `sessionStorage`: APIキーなど、セッション限りの秘密情報
- `IndexedDB`: `SaveFile`、イベントログ、チャット履歴、ターンスナップショット

公開Webでは、アクセス時は新規開始を基本にする。ただし同じブラウザ内では「前回の続き」を選べるようにする。ブラウザキャッシュ削除、プライベートブラウズ、別端末では消える前提とする。

### ファイル保存

- ユーザーが任意でシミュレーション状態をJSONファイルとしてエクスポートできる
- エクスポートしたJSONをインポートすると続きから再開できる
- File System Access API対応ブラウザでは、任意で「上書き保存」に近い体験を提供できる
- 非対応ブラウザでは、ダウンロード/アップロード方式で対応する

### 保存データに含めるもの

- 現在ターン
- 指標値
- 属性別状態
- 発生済み課題
- 生成済み意見
- 採用施策
- ランダムイベント履歴
- AI要約・結果レポート
- 乱数シード

### 保存データに含めないもの

- APIキー
- Codex/Claude Codeなどの認証情報
- 個人情報
- サーバー側秘密情報

### エクスポート/インポート

エクスポートは `SaveFile v1` のJSONとして出力する。

- ファイルには `schemaVersion` を必ず含める
- インポート時はJSON Schemaで検証する
- 未知のschemaVersionは読み込まず、エラー表示する
- 互換更新が必要になった場合はマイグレーション関数を用意する
- APIキー、Codex認証、Claude Code認証、Providerの秘密情報は含めない
- インポート後は、保存ファイル内の `memorySummary` と `timeline` から次ターンを再開する

保存ファイルは「その時点の状態」と「過去の推移」を両方持つ。これにより、単に再開できるだけでなく、年度末レポートや次ターンAI入力に過去の文脈を使える。

## 9. シミュレーションエンジン

AIの文章生成と、指標計算は分離する。

- AI: 意見・論点・改善案・施策結果シミュレーション・レポートの生成
- ルールエンジン: AI出力の検証、指標反映、予算制約、キャッシュ制約、ランダムイベント反映幅の制御

これにより、AIの出力が毎回変わっても、ゲームとしての一貫性を保つ。

### 予算とキャッシュ

予算とキャッシュは別指標として扱う。

- 予算: 今学期または年間で割り当てられている総枠
- キャッシュ: 生徒会が比較的自由に使える手元資金

基本的に施策はキャッシュの範囲内で実施する。ただし、他の予算枠を充当する、次学期予算を前倒しする、外部支援を得るなどの案もあり得る。

その場合、AIとチャットで財源案を検討し、以下の副作用も評価する。

- 他活動へのしわ寄せ
- 将来ターンのキャッシュ減少
- 特定属性からの不公平感
- 教師・学校側の承認ハードル
- 内部スコアへの影響

### ガードレール

初期MVPでは生徒の声を中心にする。ただし、生徒が自由に提案できる範囲と、学校運営として認めにくい範囲を分ける。

ガードレールはAI任せにしない。ルールベースの判定とAIによる理由説明を併用する。

- 生徒は自由に不満・要望・提案を出せる
- 一般に学校運営上認めにくい施策は、採用不可または要修正として扱う
- 危険行為、差別、いじめ助長、違法行為、個人攻撃、過度なプライバシー侵害、教育機会の不当な制限はNG
- NGの場合も、単に却下せず「背景にあるニーズ」を抽出し、代替施策へ変換する
- 教師・保護者は初期MVPでは主要発話者にしないが、ガードレール判定や副作用評価の観点として登場させる

例:

```text
元提案: 特定の生徒を文化祭から排除する
判定: 採用不可
背景ニーズ: 迷惑行為への不安、イベント運営の安全性確保
代替案: 参加ルールの明文化、トラブル時の相談導線、当日運営スタッフの増員
```

### 年度末レポート

1年間が終わると、見えるスコアと見えないスコアを総合した運営レポートを出す。

年度末レポートで表示する内容:

- 1年間の採用施策
- 見える指標の変化
- 内部スコアを加味した総合評価
- 短期的に成功したが長期的に副作用があった点
- 少数派への影響
- 教師・保護者など外部ステークホルダー視点の懸念
- 次年度への引き継ぎ課題

このレポートにより、ターン中には見えなかった内部評価が後から効いてくる構造にする。

### 初期状態生成

初期状態は固定JSONと乱数シードから生成する。

- 指標定義はJSONで固定管理する
- 各指標は `base` と `variance` を持つ
- ペルソナ、属性、学校設定もJSONで管理する
- AIは初期指標を勝手に変更しない
- AIは与えられた状態を解釈して、発言・課題候補・改善案を生成する

### 課題選定

課題は最初から固定しない。

1. ルールエンジンが学園状態を作る
2. AIがペルソナ発話を生成する
3. AIが発話から課題候補を抽出する
4. 生徒会プレイヤーがAIチャットで課題候補を深掘りする
5. 生徒会プレイヤーが課題を選ぶ
6. 選ばれた課題に対して施策検討チャットを開始する

これにより、同じアプリでも開始ごとに違う学園状態と論点が出る。

### 施策選定

1学期に実施する施策は1つとする。

ただし、1つの施策が複数の課題に対応することは許容する。たとえば「放課後学習支援と自習室運用改善をセットにした施策」は、成績低下、自習室不足、経済格差の複数課題に同時に影響してよい。

施策案には、主対象課題と副次的に対応する課題を明示する。

### 施策データの基本形

```json
{
  "id": "casual_uniform_twice_week",
  "title": "週2日の私服登校制度",
  "cost": 10,
  "shortTermEffects": {
    "studentSupport": 8,
    "happiness": 5,
    "teacherSupport": -2,
    "inequality": 3
  },
  "groupEffects": {
    "freedom_oriented": 12,
    "rule_oriented": -3,
    "low_income": -4
  },
  "longTermEffects": {
    "schoolReputation": 2,
    "teacherLoad": 2
  },
  "risks": ["inequality_visibility", "rule_management_load"]
}
```

## 10. 画面構成案

- 簡易ダッシュボード: 生徒会から見える指標、予算、アンケート結果、成績、参加率
- 財務表示: 予算、キャッシュ、施策費用、予算流用の有無
- 詳細ダッシュボード: 属性別反応、推移、課題マップ、内部スコアの存在
- 声の収集: ペルソナ別の意見、発言量、感情、属性
- 論点整理: クラスター、対立軸、少数意見、論点マップ
- 課題選択チャット: 課題候補の深掘り、優先順位付け、今学期の課題確定
- 改善案検討チャット: AI提案、ユーザー編集、予算配分、財源案、リスク表示
- 意思決定: 施策選択、投票結果、合意度、採用理由
- 結果: AIシミュレーション、予測と実績の差分、指標変化、属性別影響、ランダムイベント、次ターン論点
- 年度末レポート: 見えるスコアと見えないスコアを加味した1年間の評価
- 履歴: 過去ターンの決定と長期影響
- 保存/読込: JSONエクスポート、JSONインポート、ブラウザ内保存のクリア

## 11. データ設計の主要エンティティ

- `SimulationState`: 現在ターン、予算、キャッシュ、全体指標、履歴
- `Metric`: 指標定義、現在値、増減、警戒ライン
- `VisibleMetric`: 生徒会ダッシュボードに表示する指標
- `HiddenScore`: 通常は表示しない内部評価
- `Group`: 属性グループ、人数比、価値観、反応係数
- `Persona`: 仮想生徒・教師・保護者の人物像
- `Issue`: 課題イベント
- `IssueCandidate`: AIが抽出した課題候補
- `IssueSelectionChat`: 課題候補を深掘りするチャット履歴
- `Opinion`: ペルソナ発言
- `DeliberationSummary`: AIによる論点整理
- `PolicyOption`: 施策案
- `Decision`: 採用施策と予算配分
- `FinancePlan`: キャッシュ利用、予算流用、財源案
- `AiSimulationResult`: AIによる施策結果シミュレーション
- `TurnResult`: 予測、実績、ランダムイベント、レポート
- `AnnualReport`: 1年間の見える成果と内部評価を統合した年度末レポート
- `SaveFile`: エクスポート可能なシミュレーション保存データ
- `AiProviderConfig`: AI接続モードと公開/ローカル制約
- `ScenarioDefinition`: 学園版・国家版などのスキン定義
- `GuardrailResult`: 提案の採用可否、理由、代替案

## 12. 技術方針

初期実装はローカルで動くWebアプリを優先する。

- Frontend: Next.js / React
- UI: Tailwind CSSまたはshadcn/ui
- Visualization: Recharts
- API: Next.js Route Handlers
- DB: MVPでは使わない。ブラウザ内保存とJSONファイル保存を基本にする
- AI: Provider抽象化レイヤー。OpenAI固定にはしない
- デプロイ候補: VercelまたはCloudflare Pages

MVPではサーバーDBを持たず、まずはJSONシナリオ、ブラウザ内保存、ファイル保存でゲームループを成立させる。

### 初期JSONデータ

最初は以下をJSONで持つ。

- 学園設定
- 指標定義
- 属性グループ
- ペルソナ雛形
- 予算ルール
- 見えるスコアと見えないスコアの定義
- ランダムイベント定義
- 施策効果テンプレート
- AIプロンプトテンプレート

ユーザーの進行データはJSONファイルとしてエクスポートできるが、初期固定データとは分ける。

### 初期ディレクトリ案

```text
data/
  scenarios/
    school/
      scenario.json
      metrics.json
      groups.json
      persona-templates.json
      random-events.json
      policy-effect-templates.json
      prompt-templates.json
```

### AI入出力の最小スキーマ

初期MVPでは、AI処理を以下の4種類に分ける。

```text
generateOpinions
入力: SimulationState, ScenarioDefinition, PersonaTemplate[]
出力: Opinion[]

extractIssueCandidates
入力: SimulationState, Opinion[]
出力: IssueCandidate[]

discussIssueSelection
入力: SimulationState, Opinion[], IssueCandidate[], PolicyChatMessage[]
出力: IssueSelectionChatResponse

generatePolicyOptions
入力: SimulationState, IssueCandidate, Opinion[]
出力: PolicyOption[]

evaluateGuardrails
入力: SimulationState, PolicyOption
出力: GuardrailResult

generateTurnReport
入力: SimulationState, Decision, TurnResult
出力: TurnResultReport

simulatePolicyResult
入力: SimulationState, Decision, PolicyChatMessage[]
出力: AiSimulationResult
```

AIが返す値は文章だけにしない。必ずID、関連指標、関連属性、リスク、根拠を含む構造化データとして返す。

### 課題候補データ例

## 13. 仮想国家版への拡張方針

学園固有名をコアエンジンに埋め込まない。

- 学園版: 生徒会費、生徒支持率、教師支持率、学校評判、成績、校則
- 国家版: 国家予算、国民支持率、官僚・専門家支持率、国際評価、国力、法制度

コアは `civic simulation engine` として、指標セット、属性セット、イベントセット、施策セットを差し替える。

### 抽象設計の方針

最初から「学園版はシナリオスキン」として扱う。

- `core`: ターン進行、指標計算、乱数、保存、AI入出力検証
- `scenario/school`: 学園版の指標、属性、ペルソナ、文言、イベント、施策テンプレート
- `scenario/nation`: 将来の仮想国家版の差し替え先

ただし、初期実装で過度に汎用化しない。まず学園版を動かし、そのうえで国家版に必要な共通点をコアへ移す。

## 14. 最初に確認したい仕様論点

### 決定済み

- 1ターンは1学期
- プレイヤーは生徒会
- 初期課題は固定せず、初期指標とペルソナ発話からAIが候補を抽出する
- 初期固定データはJSON
- 仮想国家版を見据え、学園版はシナリオスキンとして抽象設計する
- 公開WebのAIキー運用は `bring-your-own-key`
- 運営側はAPIキーを提供しない
- ローカルAI接続の `codex-cli` は正式対応候補に含めるが、初期MVP必須にはしない
- 1学期に実施する施策は1つ
- 1つの施策が複数課題に対応することは許容する
- 初期MVPのペルソナ発話は生徒中心
- 教師・保護者はガードレールや副作用評価の観点として扱う
