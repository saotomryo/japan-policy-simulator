# 開発計画

## 目的

学園自治シミュレーターは、AIによる仮想データ生成、課題抽出、分析、対話的な施策検討を中核にする。

AIなしで成立するゲームではなく、AIを通じて「多様な声を再現し、論点化し、チャットで深掘りし、施策を作る」アプリとして設計する。

最初のゴールは、1学期分のAI中心ループをローカルで体験できること。

```text
初期状態生成
↓
AIが生徒ペルソナ発話を生成
↓
AIが課題候補を抽出
↓
課題選択チャットで候補を深掘り
↓
ユーザーが課題を確定
↓
AI分析結果を表示
↓
チャットで分析・候補案を深掘り
↓
AIと一緒に施策案を確定
↓
ガードレール判定
↓
AIが施策結果をシミュレーション
↓
アプリが指標変更を検証・反映
↓
結果と年度末評価へ反映
```

## 前提

- 公開WebのAIキー運用は利用者自身のAPIキー入力
- 運営側APIキーは提供しない
- APIキーは保存データに含めない
- AI ProviderはOpenAI固定にしない
- 初期実装では `sample`、`openai_responses`、`openai_compatible_chat` から始める
- ローカル実行では `codex_app_server` Providerを選べるようにする
- AI処理はアプリ体験の中心
- 課題選択の時点からチャットを使う
- 施策結果のシミュレーションにもAIを使う
- `local-mock` はAIなしで遊ぶためではなく、開発中の固定サンプル、テスト、障害時fallbackに限定する
- 現在のCodex上で仮想データや分析例を作ることは、開発初期のAI代替処理として扱える
- ローカル `codex-cli` 連携は `codex app-server` を優先する
- `codex app-server` は公開Webでは無効化し、ローカルビルドまたは開発時オプションとして扱う
- `codex app-server` を使う場合、アプリとは別に `npm run codex:app-server` を起動し、ProviderのBase URLを `ws://127.0.0.1:45123` にする

## Phase 4: 分析結果と施策検討チャット（MVP完了）

目的: 施策案を単発生成ではなく、分析結果を見ながらチャットで深掘りして作る。

### 現状

MVPとしては完了扱いにする。

- 選択課題、現在指標、財務、季節イベント、属性メモリーを文脈に入れて施策ドラフトを生成できる

- 施策検討チャットから、現在の施策案全体を更新できる
- AI接続が失敗した場合も固定サンプルにfallbackして流れを継続できる

独立した「分析レポート画面」はまだ明確には分かれていない。現状は、課題選択チャット、声の分析、施策ドラフト生成プロンプト内の文脈、施策案表示が分析結果の役割を兼ねている。必要になった段階で、分析レポートを独立カードとして追加する。

kouchou-aiなどで出力された、カテゴライズ済みの意見データを取り込み、声の分析、課題設定、施策立案につなげる。

想定する初期対応フォーマットはCSVとする。例として確認したkouchou-ai CSVは以下の列を持つ。

- `comment-id`: 元コメントID
- `original-comment`: 元の意見本文
- `arg_id`: 抽出・整理された論点ID
- `argument`: 抽出・整理された論点文
- `category_id`: カテゴリID
- `category`: カテゴリ名またはカテゴリ要約
- `source`: データソース
- `url`: 参照URL

確認したサンプルでは、1168件のコメント、360件の論点、10カテゴリが含まれていた。

#### 取り込み後の扱い

- `original-comment` を個別の声として扱う
- `argument` を代表論点として扱う
- `category` を意見クラスターまたは課題候補の種として扱う
- `category_id` と `arg_id` を分析・掘り下げ時の参照IDとして保持する
- `source` と `url` を出典情報として保持する
- 取り込みデータから代表発話を抽出し、声の分析画面に表示する
- カテゴリ別件数、代表論点、類似論点、少数意見を可視化する
- 課題設定画面では、外部データ由来の課題候補をAIに生成させる
- 施策検討では、外部意見データを根拠資料としてAI文脈に渡す

#### UI案

- 保存画面または専用の「データ取り込み」画面にCSVインポートを追加する
- 取り込み前に列マッピングを確認できるようにする
- kouchou-ai形式はプリセットとして自動認識する
- 取り込み後に、件数、カテゴリ数、論点数、代表カテゴリをプレビューする
- 取り込みデータを現在の仮想生徒の声へ混ぜるか、外部資料として別枠で扱うかを選べるようにする
- 既存の仮想データを置き換える、新規ターンから反映する、課題設定だけに使う、という反映範囲を選べるようにする

#### データ設計案

- `externalOpinionSources`: 取り込み元ファイル、形式、件数、取り込み日時
- `externalOpinions`: 元コメント単位のデータ
- `externalArguments`: 論点単位のデータ
- `externalCategories`: カテゴリ単位の集計データ
- `voiceAnalysis.sourceMode`: `generated`, `external`, `mixed`
- 保存ファイルとJSONエクスポートに外部データ由来の分析結果を含める

#### 完了条件

- kouchou-ai形式CSVを読み込める
- 列マッピングと取り込み結果を確認できる
- 外部意見データから声の分析画面を構成できる
- 外部意見データを課題候補生成と施策検討チャットの文脈に渡せる
- 外部データを含む状態を保存・エクスポート・インポートできる

### 第二フェーズの完了条件

- 初期データ生成が、設定に基づいて説明可能な形になる
- ユーザーが画面上でシミュレーション前提を変更できる
- 外部意見データを取り込み、声の分析から課題設定、施策立案へつなげられる
- プロンプトを編集して生成方針を調整できる
- 施策実行前にガードレール判定を確認できる
- AI接続失敗時に、どこで何が起きたか分かる
- 第一フェーズの保存データを壊さずに読み込める
