# 国版 政策シミュレーター 開発計画

## 1. 開発方針

まずはモックデータで国版の画面導線を実際に動かす。

AI連携、実統計値の精査、JSON Schemaの厳密化は重要だが、先に画面上で `政策ターゲット -> 声の分析 -> 政策案 -> 実行結果` の流れを触れる状態にする。画面で確認しながら、指標・効果軸・クラスター表示の粒度を詰める。

### 優先すること

- ローカルで起動してすぐ確認できること
- 固定モックデータでも一連の体験が成立すること
- 指標や効果軸を固定表示にせず、政策ごとの関連指標・推奨ビューに切り替えられること
- 小さく実装し、毎フェーズでブラウザ確認すること
- 既存の保存、AI設定、Provider設計を壊さず段階的に置き換えること

### 後回しにすること

- 実統計値のオンライン更新
- 本格的なAIプロンプト調整
- 政策分野分類の固定化
- 複数ターン運営
- 財源制約による強制失敗
- 長期結果の確定シミュレーション

### 将来検討メモ: 静的版とローカル深掘り版の二層構成

現時点では、GitHub Pagesなどで静的に動く構成を維持し、まずは画面、固定モック、ブラウザ内AI接続、保存、HTMLレポート出力をブラッシュアップする。

一方で、声の生成、クラスター分析、政策案作成は、現状では実データに強く基づくというより、固定初期値とAI推論による仮説生成に近い。将来的には、cloneしてローカル実行する場合だけ、Pythonなどのバックエンドプロセスを追加し、より深い分析モードを選べる構成を検討する。

想定する構成。

```text
GitHub Pages / 静的公開版
- UI確認
- 固定モック
- ユーザーAPIキーまたはCodexローカルブリッジによる簡易AI生成
- JSON保存
- 単一HTMLレポート出力

ローカル拡張版
- Pythonバックエンド起動
- Web検索
- 公的統計、白書、PDF、CSVの取得と要約
- エージェント型生成
- 内部レビュー
- 出典付きレポート
- 中間結果キャッシュ
```

将来的な生成モード案。

- `static_mock`: 高速モック。UI確認と仕様検討を優先
- `browser_ai`: ブラウザからAI Providerへ直接問い合わせる簡易生成
- `local_codex_bridge`: Codex App Server経由のローカルAI生成
- `web_researched`: 検索・公的資料参照ありの生成
- `agent_reviewed`: 調査、生成、財政レビュー、世論妥当性レビュー、業界影響レビューを経由する生成

### 将来検討メモ: 専門家レビューを追加プロセスとして実行する

通常の生成フローは、操作速度と試行錯誤のしやすさを優先し、以下を維持する。

```text
政策ターゲット指定 -> 声の生成 -> クラスター分析 -> 初期政策案 -> 実行結果
```

一方で、実際の政策議論との比較、因果経路の妥当性、効果の過大評価・過小評価は、通常生成プロンプトだけに混ぜると処理時間が長くなり、通常操作のテンポも悪くなる。そのため、政策案画面または実行結果画面に `専門家レビュー` / `実際の政策議論と比較` / `政策効果の妥当性を検証` のような追加ボタンを置き、必要なときだけ実行する追加プロセスとして扱う。

専門家レビューで確認する観点。

- 政策目的の分類。例: 家計支援、産業政策、財政再建、規制改革、安全保障、地域政策、資産形成政策
- 実際に議論されている類似政策・政策争点との比較
- 政策効果の因果経路。直接効果と間接効果を分け、因果の強さを評価する
- 各指標変化の妥当性。妥当、やや過大評価、やや過小評価、根拠不足、政策設計が曖昧で判定困難に分ける
- 所得別、世代別、産業別、財政影響、国際関係、実装リスクの妥当性
- 国民の声・クラスター構成の現実味
- 政策案の制度設計上の弱点
- 修正すべき指標値、説明文、追加すべき論点
- 追加で確認すべきデータ、公的資料、統計、政策文書

レビュー専用プロンプトの方針。

- アプリ仕様への適合ではなく、政策専門家・評論家の観点でレビューする
- 政策を擁護しすぎず、反対論だけにも寄せない
- 効果経路を明示し、直接効果と間接効果を混同しない
- 実データに基づかない判断は推測として扱う
- 評価できないものは無理に数値化せず、根拠不足または設計不明として返す
- 例: NISA関連政策では、流通市場での株式購入と企業への新規資金供給を混同しない。国内株購入、IPO、公募増資、スタートアップ投資、企業統治改善、金融教育、海外投資偏重、円安、租税支出の論点を分ける

構造化出力の候補。

```js
expertReview: {
  overallReview: "",
  policyType: ["資産形成政策", "産業政策"],
  realWorldComparison: [
    {
      topic: "資産運用立国",
      similarity: "高い" | "中" | "低い",
      comment: ""
    }
  ],
  causalPathways: [
    {
      pathway: "",
      strength: "強い" | "中" | "弱い" | "根拠不足",
      reason: ""
    }
  ],
  metricReviews: [
    {
      metricId: "",
      label: "",
      currentDelta: 0,
      judgement: "妥当" | "やや過大評価" | "やや過小評価" | "根拠不足" | "判定困難",
      suggestedDelta: 0,
      reason: ""
    }
  ],
  recommendedRevisions: []
}
```

初期実装では、インターネット検索なしの軽量レビューを想定する。将来的にローカル拡張版またはVercel/Pythonバックエンドを持つ場合、検索・公的資料・統計参照ありの詳細レビューを別モードとして追加する。

本格的な検索あり生成やエージェント型生成では、以下の理由からバックエンドを持つ方が現実的。

- 長時間処理と進捗管理が必要
- Web検索、PDF、CSV、統計データ処理が必要
- 中間結果、出典、キャッシュを管理したい
- コスト上限、リトライ、失敗時復旧が必要
- 内部レビュー結果を保存し、レポートに反映したい

将来のデータ構造候補。

```js
evidence: {
  sources: [],
  assumptions: [],
  dataQuality: "mock" | "ai_estimated" | "web_researched" | "official_stat",
  confidenceScore: 0
}

reviews: {
  fiscalReview: [],
  publicOpinionReview: [],
  industryReview: [],
  consistencyWarnings: []
}
```

ただし、当面の優先順位は静的実装の品質向上とする。検索あり生成、Pythonバックエンド、エージェント型内部レビューは、UIとデータ契約が安定してから検討する。

## 2. MVP到達イメージ

MVPでは、ユーザーが以下をローカルブラウザで操作できる状態を目指す。

1. 国版ダッシュボードを見る
2. プリセット政策を選ぶ
3. 政策ターゲットの構造化結果を見る
4. 関連指標・推奨ビューが政策に応じて切り替わる
5. 国民・ステークホルダーの声を見る
6. 世代別、所得別、地域別、国際関係などの効果軸を切り替える
7. 政策案を見る
8. チャット風UIで分析観点を切り替える
9. 政策を1回だけ実行する
10. 短期結果と属性別影響を詳細レポートで見る
11. 保存、JSONエクスポート、JSONインポートができる

## 7. リスクと対策

### 学園版の概念が残る

対策: まず画面文言とデータキーを国版に寄せる。内部互換のために一時的に `issues` を残す場合も、画面には出さない。

### データ構造を一気に変えすぎて壊れる

対策: Phase 1では互換キーを残して動かし、Phase 4でスキーマと保存形式を整理する。

### AI連携前に仕様が固まりすぎる

対策: sample Providerで操作感を確認し、効果軸や表示粒度を画面で詰めてからAIプロンプトに反映する。

### 指標表示が政策に合わない

対策: `relatedMetricIds` と `recommendedViews` を中核にし、政策ごとに初期表示を変える。

### 実統計値の扱いが重くなる

対策: MVPでは固定初期値に限定する。出典・基準年・データ種別を持たせ、オンライン更新は後回しにする。
